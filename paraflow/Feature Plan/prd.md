# PRD — SOLVISTA MVP（社会課題構造化プラットフォーム）

## 1) 背景

大学研究機関や企業の新規事業開発部門では、社会課題解決をテーマに掲げる際、「解決すべき課題が明確に定義できない」という根本的な問題に直面しています。顧客開発モデルやFITジャーニーは「課題が既に言語化されている」前提で設計されており、その前段階である「課題の特定と構造化」をサポートする専用ツールが存在しませんでした。Product Charterで定義した「構造化思考」「反復的洗練」「AI駆動の洞察」という価値提案を実現するため、プライマリーペルソナである山田啓介（研究主導型リーダー）とセカンダリーペルソナの佐藤美咲（イノベーション推進担当）が、複数のステークホルダーと協働しながら課題を体系的に言語化できる環境を提供します。今こそこの課題に取り組むべき理由は、SDGsやCSVへの社会的関心の高まりにより、社会課題解決型プロジェクトの数が急増している一方、その多くが実装に至らず停滞しているためです。

## 2) 目標と期待される成果

- **成果A:** ユーザーが曖昧だった社会課題を、ミッションモデルキャンバスのフレームワークに沿って構造化し、チーム全員が同じ認識を持てるようになる
- **成果B:** AI分析によるケーススタディベースのフィードバックを受けることで、思考の盲点が減り、課題定義の質が飛躍的に向上する
- **成果C:** 複数のステークホルダーが非同期で共同編集・議論できる環境により、合意形成のスピードが向上し、社会実装までの期間が短縮される
- **成果D:** すべての編集履歴と意思決定プロセスが可視化され、助成金申請や経営陣への報告時に説明責任を果たせる
- **対象外 / 境界:** ミッションモデルキャンバスの残り5項目（ソリューション、主要指標、独自の価値提案、チャネル、圧倒的な優位性）の言語化は、別プラットフォーム「SMATSTA」で扱うため、本MVPでは対象外とする。また、リアルタイムビデオ会議機能やチャット機能は外部ツールとの連携で対応し、本体には含めない

## 3) ユーザーとストーリー

- **プライマリーペルソナ:** 山田啓介（研究主導型リーダー）
  - ストーリーA: 研究者として、新しい社会課題プロジェクトを構造化されたフレームワークで定義したい。そうすることで、チームメンバーや外部パートナーとの認識のズレを最小化できる
  - ストーリーB: 研究者として、ステークホルダーごとの課題（ゴール・現状・ギャップ）を段階的に言語化したい。そうすることで、属人的な解釈に依存せず、体系的に課題を整理できる
  - ストーリーC: 研究者として、AI分析からの客観的なフィードバックと深掘り質問を受けたい。そうすることで、自分たちだけでは気づけなかった視点や関連課題を発見できる
  - ストーリーD: 研究者として、過去のバージョンを参照しながら課題定義の進化を振り返りたい。そうすることで、プロジェクトの意思決定プロセスを可視化し、助成金申請書に反映できる
- **セカンダリーペルソナ:** 佐藤美咲（イノベーション推進担当）
  - ストーリーE: 新規事業開発担当として、複数のオープンイノベーションプロジェクトを一覧で管理したい。そうすることで、各プロジェクトの進捗状況を素早く把握できる
  - ストーリーF: 新規事業開発担当として、外部パートナーと非同期で課題定義を共同編集したい。そうすることで、オンライン中心のコミュニケーションでも合意形成をスムーズに進められる
  - ストーリーG: 新規事業開発担当として、完成した課題定義をJSON形式でエクスポートしたい。そうすることで、社内の事業計画書フォーマットや次フェーズのツール（SMATSTA）にシームレスに連携できる

## 4) 主要機能

- **プロジェクト管理とダッシュボード:** ユーザーは複数の社会課題プロジェクトを作成・管理し、各プロジェクトの名前・説明・ステータスを一覧表示できる。プロジェクトごとに編集履歴が自動的にバージョン管理され、過去のバージョンにいつでも戻れる
- **課題の構造化言語化（ステークホルダー課題・社会システム課題）:** ユーザーは課題タイプ（ステークホルダー課題 or 社会システム課題）とカテゴリー（クライシス/リスク/問題/課題）を選択し、タイトル・説明・ゴール・現状・ギャップを段階的に記入できる。1つのプロジェクト内で複数のステークホルダーや社会システムを扱える
- **AI分析と客観的フィードバック:** ユーザーが課題を言語化すると、AIがケーススタディデータベースを参照し、入力内容を分析する。AIは客観的なアドバイス、関連課題、影響範囲、潜在的ステークホルダー、深掘り質問を提示する
- **共同編集とチームコラボレーション:** ユーザーは同じプロジェクトにチームメンバーやパートナー組織を招待し、リアルタイムで課題定義を共同編集できる。メモ機能で議論を記録し、投票機能で合意形成を促進する
- **バージョン管理と履歴表示:** すべての編集がバージョンとして記録され、ユーザーは任意のバージョンを閲覧・比較・復元できる。変更履歴により、誰がいつ何を変更したかが透明化される
- **JSONエクスポートと外部連携:** ユーザーは完成した課題定義をJSON形式でエクスポートし、他ツール（SMATSTA、スプレッドシート、事業計画書テンプレート）にインポートできる

## 5) 主要フロー

- **新規プロジェクト作成から課題言語化まで:**
  - **トリガー:** 研究者や新規事業担当者が新しい社会課題解決プロジェクトを開始する
  - **経路:** ユーザーはダッシュボードから「新規プロジェクト作成」をクリックし、プロジェクト名と説明を入力する。次に「課題を追加」から課題タイプ（ステークホルダー課題 or 社会システム課題）とカテゴリーを選択し、タイトル・説明・ゴール・現状・ギャップを段階的に記入する。システムは入力内容を自動保存し、バージョン1として記録する
  - **結果:** ユーザーは構造化されたフレームワークに沿って課題を言語化でき、チーム全員が同じフォーマットで課題を理解できる状態が実現する

- **AI分析を活用した課題のブラッシュアップ:**
  - **トリガー:** ユーザーが課題の初期言語化を完了し、客観的なフィードバックを求める | **経路:** ユーザーは「AI分析を実行」ボタンをクリックする。AIはケーススタディデータベースを参照し、入力内容に対する客観的なアドバイス、関連課題、影響範囲、潜在的ステークホルダー、深掘り質問を生成する。ユーザーはAIの提案を参考に課題定義を修正し、システムは新しいバージョンとして保存する | **結果:** ユーザーは思考の盲点に気づき、課題定義の解像度が向上する

- **共同編集による合意形成:**
  - **トリガー:** 複数のステークホルダーが異なる視点を持ち、課題定義の合意が必要な状況 | **経路:** ユーザーはチームメンバーやパートナー組織をプロジェクトに招待し、共同編集権限を付与する。各メンバーは課題定義を編集・コメント・投票でき、システムはすべての変更を記録する。議論が収束したら、ユーザーは最終バージョンを確定する | **結果:** ステークホルダー全員の意見が可視化され、透明性の高い合意形成が実現する

- **JSONエクスポートと次フェーズへの連携:**
  - **トリガー:** 課題定義が完了し、次のフェーズ（ソリューション開発やSMATSTAへの移行）に進む準備ができた | **経路:** ユーザーは「エクスポート」ボタンをクリックし、JSON形式を選択する。システムは構造化されたデータを生成し、ユーザーはファイルをダウンロードする | **結果:** 課題定義データが他ツールにシームレスに連携され、手作業での転記が不要になる

## 6) 競合分析

- **ランドスケープ（誰がこの問題を解決しているか）:**
  - **Miro / Mural（オンラインホワイトボード）:** デザイン思考ワークショップやブレインストーミングを行うチーム向け
  - **Notion / Confluence（ドキュメント管理）:** 汎用的なドキュメント作成と共同編集を求める組織向け
  - **Google Docs / スプレッドシート（汎用ツール）:** 無料で手軽に始めたい小規模チーム向け
  - **専用フレームワークなし（手作業）:** 既存ツールに依存せず、Word文書やPowerPointで独自に整理するチーム向け
  - **コンサルティングサービス:** 外部専門家の伴走支援を求める大企業や自治体向け

- **価値提案（各プレイヤーの提供価値）:**
  - **Miro / Mural:** 自由度の高いビジュアル思考とリアルタイムコラボレーションを提供するが、構造化されたフレームワークは提供せず、整理は利用者に依存する
  - **Notion / Confluence:** 柔軟なドキュメント作成と知識管理を提供するが、社会課題解決に特化したテンプレートやAI分析はなく、ゼロから設計する必要がある
  - **Google Docs:** 無料で即座に始められるが、バージョン管理や構造化思考のサポートが弱く、属人的な整理に陥りやすい
  - **手作業:** 完全に自由だが、フレームワークもフィードバックもなく、非効率で再現性が低い
  - **コンサルティング:** 専門家の知見と伴走支援が得られるが、コストが高く、内製化が困難

- **強み / 弱み（体験の利点と欠点）:**
  - **Miro / Mural:** オンボーディングは直感的だが、自由度が高すぎて初心者は迷いやすい。編集可能性は高いが、構造化されたアウトプットを得るには手間がかかる
  - **Notion / Confluence:** ドキュメント作成の自由度は高いが、社会課題解決に特化した指針がなく、テンプレートを自作する必要がある。拡張性は高いが、AI分析機能はない
  - **Google Docs:** 即座に使えるが、複雑なプロジェクト管理には不向き。共同編集は可能だが、バージョン管理や合意形成機能が貧弱
  - **手作業:** 完全に自由だが、客観性や再現性がなく、ステークホルダー間の認識ズレが解消しにくい
  - **コンサルティング:** 専門家の知見が得られるが、依存度が高く、内製化や横展開が困難

- **私たちの差別化要素（独自の強み）:**
  - SOLVISTAは「構造化思考」「反復的洗練」「AI駆動の洞察」「協働の場」「透明性」というブランドキーワードを体現し、社会課題解決の「前段階」に特化した唯一のプラットフォームです
  - ミッションモデルキャンバスの重要4項目に絞り込むことで、ユーザーは迷わず段階的に課題を言語化できます
  - ケーススタディベースのAI分析により、客観的なフィードバックと深掘り質問を提供し、思考の盲点を減らします
  - バージョン管理と共同編集機能により、反復的な改善プロセスが自然に実践され、ステークホルダーとの合意形成がスムーズになります
  - トレードオフとして、SOLVISTAは汎用的なドキュメント作成ツールではなく、社会課題解決に特化した専用ツールであるため、他の用途には向きません

- **移行コストとリスク（既存ソリューションからの乗り換え）:**
  - 既存のMiroやNotionで管理していたプロジェクトデータをSOLVISTAに移行する際、手作業での転記が必要になる可能性がある（初期MVPではインポート機能は限定的）
  - ユーザーが「自由度の高いツール」に慣れている場合、構造化されたフレームワークを「制約」と感じるリスクがある
  - AI分析の提案を過度に信頼し、自分たちの判断を放棄してしまう誤用リスクがある（AIは「示唆」であり、最終判断は人間が行うべき）
